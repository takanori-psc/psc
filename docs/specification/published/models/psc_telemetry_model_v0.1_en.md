## 1. Purpose

This document defines the Telemetry Model in the PSC Fabric.

The Telemetry Model observes, normalizes, and evaluates Fabric state,
and converts it into routing-consumable operational state.

The model provides the following outputs:

- congestion_score
- health_state
- availability_state
- trust_score_ref (optional)

Telemetry is not merely an observation system, but a
**state generation model for routing and control**.

## 3. Design Principles

### 3.1 Separation of Responsibilities

- Telemetry: state generation
- Routing Table: state storage
- RCU: decision making

Responsibilities must remain strictly separated.

### 3.2 Observability

Telemetry shall be based on observable data and referenced inputs.

### 3.3 Stability over Noise

Persistent conditions are prioritized over transient fluctuations.

### 3.4 Routing-facing Simplicity

Outputs must be directly consumable by routing components.

---

## 4. Telemetry Entry Model

Telemetry is represented per evaluation target as:

```text
TelemetryEntry {
    telemetry_id
    target_type
    target_id
    scope_level
    timestamp
    input_snapshot
    evaluated_state
    update_reason
    confidence
    ttl
}
```

### Field Definitions

- telemetry_id
  Unique identifier of the telemetry entry

- target_type
  Target type (LINK / NODE / PATH / FABRIC)

- target_id
  Identifier of the target

- scope_level
  Evaluation scope

- timestamp
  Evaluation timestamp

- input_snapshot
  Input values used for evaluation

- evaluated_state
  Output state values

- update_reason
  Reason for update

- confidence
  Confidence level of evaluation

- ttl
  Validity duration

## 5. Input Parameters

Telemetry uses the following inputs.

### 5.1 Observed Inputs

#### Link-related

- link utilization
- latency
- retry count
- packet loss
- error rate
- credit starvation
- link state (up/down)

#### Node-related

- node state
- processing backlog
- queue occupancy
- buffer pressure
- retry concentration
- timeout events

#### Queue / Buffer

- queue depth
- queue saturation ratio
- buffer occupancy
- sustained backlog duration

### 5.2 Referenced Inputs

#### Policy-related

- policy constraints
- routing class requirements

#### Trust / Security-related

- trust classification
- attestation result
- security alert state

Trust is not generated by telemetry, but referenced.

---

## 6. Output Parameters

Telemetry provides the following outputs.

### 6.1 congestion_score

A continuous metric representing congestion level.

Example:

- range: 0.0 – 100.0
- higher = more congested

### 6.2 health_state

Internal operational condition of the target.

```text
HEALTHY
DEGRADED
FAILED
```

### 6.3 availability_state

Availability as a routing resource.

```text
AVAILABLE
RESTRICTED
UNAVAILABLE
```

### 6.4 trust_score_ref (optional)

Reference value from external trust evaluation.

---

## 7. Evaluation Units

Telemetry is evaluated at the following levels.

### 7.1 Link-level

Base unit for congestion evaluation.

### 7.2 Node-level

Represents internal node conditions.

### 7.3 Path-level

Aggregated evaluation across multiple components.

### 7.4 Fabric-level

Global state of the entire fabric.

---

## 8. Congestion Score Model

Congestion is evaluated using an additive model.

```text
congestion_score =
  utilization_score
+ buffer_pressure_score
+ latency_penalty
+ retry_penalty
+ loss_penalty
+ credit_starvation_penalty
+ persistence_modifier
- recovery_modifier
- stability_modifier
```

### Design Guidelines

- Avoid single-metric decisions
- Prefer persistent conditions
- Recovery should be conservative
- Apply hysteresis

---

## 9. Health State Evaluation

```text
if hard_failure:
    FAILED
elif persistent_degradation:
    DEGRADED
else:
    HEALTHY
```

### Examples

#### FAILED

- link down
- node unreachable
- fatal error

#### DEGRADED

- sustained latency
- repeated retries
- buffer pressure

---

## 10. Availability State Evaluation

```text
if unavailable_condition:
    UNAVAILABLE
elif restricted_condition:
    RESTRICTED
else:
    AVAILABLE
```

### Examples

#### UNAVAILABLE

- link down
- isolation
- administrative disable

#### RESTRICTED

- policy constraint
- trust limitation
- degraded capacity

---

## 11. Update Model

Telemetry supports the following update mechanisms.

### 11.1 Periodic Update

Regular interval updates

### 11.2 Event-driven Update

Triggered by events (failure, recovery, etc.)

### 11.3 Threshold Trigger

Triggered when thresholds are exceeded

### 11.4 Hysteresis

Prevents oscillation of state transitions

    health_state
    availability_state
    trust_score_ref

    ttl
